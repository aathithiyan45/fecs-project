FECS Software Design Specification
1. Project Overview
Project Name: Forest Emergency Communication System (FECS)
Purpose: FECS is an offline emergency communication system for forest and remote terrain environments. It allows a user module carried by trekkers or forest personnel to send alert messages through LoRa repeater nodes to a base station. The base station is connected to a laptop/server that runs the backend and dashboard application.
The application layer is responsible for receiving alerts, validating them, removing duplicates, storing valid records, showing them on a live dashboard, and triggering acknowledgments back to the user module through the base station.
________________________________________
2. High-Level Architecture
2.1 Physical Modules
1. User Module
•	Contains push button, GPS, LoRa, ESP32, LED, buzzer
•	Sends alert messages based on button interaction
•	Receives acknowledgment from the base station through relay nodes
2. Relay Module
•	Works as a repeater/forwarding node
•	Receives LoRa packets and retransmits them
•	Extends network coverage between user module and base station
•	Does not perform backend logic
3. Base Station Module
•	Connected to laptop/server
•	Receives messages from user modules through relay nodes
•	Forwards structured data to backend
•	Receives ACK commands from backend and transmits ACK back through relay nodes to user module
2.2 Logical Data Flow
User Module -> Relay Module(s) -> Base Station Module -> Backend -> Database -> Dashboard
Backend -> Base Station Module -> Relay Module(s) -> User Module
________________________________________
3. Application Scope
The software application includes: - Backend service using FastAPI - Frontend dashboard using React.js - Live map using Leaflet - PostgreSQL database for alert storage - MQTT-based communication between base station and backend

5. Functional Requirements
5.1 Alert Reception
The system shall receive messages from the base station module when a user module sends an alert.
5.2 Message Types
Supported message types: - normal - high - emergency - cancel
5.3 Signal Types
Supported signal types: - manual - auto
5.4 Duplicate Removal
The system shall discard duplicate messages and store only the first valid message.
5.5 Acknowledgment Trigger
The backend shall trigger ACK only for valid manual alerts: - normal - high - emergency
ACK shall not be triggered for: - auto signals - cancel messages
5.6 Dashboard Display
The dashboard shall display valid alerts on a live interface with location markers on a map.
5.7 Status Management
Each stored alert shall maintain a status value: - pending - acknowledged - resolved

6. Message Communication Design
6.1 Incoming JSON Message (Base Station -> Backend)
{
  "packet_id": "a1b2c3d4",
  "device_id": "DEV_001",
  "latitude": 10.934512,
  "longitude": 78.121245,
  "message_type": "emergency",
  "signal_type": "manual",
  "event_time": "2026-03-18T10:32:15Z",
  "source": "hardware"
}
Field Definitions
•	packet_id: unique identifier for the transmitted packet
•	device_id: identifier of user module
•	latitude: GPS latitude
•	longitude: GPS longitude
•	message_type: normal/high/emergency/cancel
•	signal_type: manual/auto
•	event_time: time generated by device
•	source: hardware or simulation
6.2 Outgoing ACK JSON (Backend -> Base Station)
{
  "device_id": "DEV_001",
  "ack_type": "delivery_ack",
  "for_packet_id": "a1b2c3d4",
  "timestamp": "2026-03-18T10:33:00Z"
}
ACK Rules
•	ACK only for valid manual alerts
•	No ACK for auto signals
•	No ACK for cancel messages
•	No retry logic
•	No delivery confirmation tracking in backend
________________________________________
7. MQTT Topic Design
Recommended simple topic structure:
fecs/incoming
fecs/ack
Topic Usage
•	fecs/incoming: base station publishes incoming alert messages to backend
•	fecs/ack: backend publishes ACK commands to base station
________________________________________
8. Duplicate Detection Rule
A message shall be treated as duplicate if all of the following are true: - same device_id - same message_type - same signal_type - event time difference is less than or equal to 10 seconds - GPS distance difference is less than or equal to 10 meters
Duplicate Handling Policy
•	store only the first valid message
•	discard all duplicate copies
•	do not store duplicate entries
________________________________________
9. Backend Processing Flow
9.1 Incoming Processing
1.	Receive JSON message from MQTT topic fecs/incoming
2.	Validate schema and field values
3.	Check for duplicate using time and distance rule
4.	If duplicate, discard message
5.	If valid, store alert in database
6.	If valid manual alert, publish ACK command to fecs/ack
7.	Push alert update to frontend dashboard
9.2 Validation Rules
•	device_id must not be empty
•	latitude must be between -90 and 90
•	longitude must be between -180 and 180
•	message_type must be one of supported values
•	signal_type must be manual or auto
•	event_time must be a valid ISO timestamp

12. Real-Time Behavior
The dashboard must update in real time when a new valid alert is stored.
Recommended implementation options: - WebSocket from FastAPI to React frontend - or periodic polling as fallback
Preferred option: WebSocket

13. Unconscious Detection Logic
Purpose
Detect whether a user may be unconscious when no manual alert is sent.
Existing Concept
•	user module sends auto location signal every 2 hours
•	backend/dashboard monitors repeated unchanged locations
•	in daytime, stagnation can escalate to high-priority alert
•	in nighttime, base station sends buzzer-trigger signal to user module
•	if user responds within configured response period, situation is treated as safe
•	if no response is received, case escalates to emergency
Current Note
This feature is part of FECS concept and can be implemented after core alert flow is completed.

16. Sequence Diagram
16.1 Alert Processing Sequence
sequenceDiagram
    participant U as User Module
    participant R as Relay Module(s)
    participant B as Base Station Module
    participant M as MQTT Broker
    participant A as FastAPI Backend
    participant D as PostgreSQL Database
    participant F as React Dashboard

    U->>R: Send LoRa alert packet
    R->>B: Forward packet
    B->>M: Publish incoming JSON to fecs/incoming
    M->>A: Deliver message
    A->>A: Validate payload
    A->>A: Check duplicate rule
    alt Valid non-duplicate message
        A->>D: Store alert record
        D-->>A: Insert success
        A->>F: Push real-time alert update
        alt Manual alert (normal/high/emergency)
            A->>M: Publish ACK command to fecs/ack
            M->>B: Deliver ACK command
            B->>R: Transmit ACK into LoRa network
            R->>U: Forward ACK
            U->>U: Blink LED
        else Auto signal or cancel
            A->>A: No ACK triggered
        end
    else Duplicate message
        A->>A: Discard message
    end
16.2 Unconscious Detection Sequence
sequenceDiagram
    participant U as User Module
    participant R as Relay Module(s)
    participant B as Base Station Module
    participant M as MQTT Broker
    participant A as FastAPI Backend
    participant D as PostgreSQL Database
    participant F as React Dashboard

    U->>R: Send auto location signal (2-hour interval)
    R->>B: Forward packet
    B->>M: Publish incoming JSON to fecs/incoming
    M->>A: Deliver message
    A->>A: Validate and store auto signal
    A->>D: Save alert/activity record
    A->>A: Compare with recent location history
    alt Same-location pattern suggests stagnation
        A->>F: Raise stagnation warning
        alt Night-time check
            A->>M: Publish buzzer-trigger ACK/command
            M->>B: Deliver command
            B->>R: Transmit through LoRa network
            R->>U: Forward buzzer trigger
            U->>U: Activate buzzer
            alt User responds in allowed time
                U->>R: Send normal alert response
                R->>B: Forward response
                B->>M: Publish to fecs/incoming
                M->>A: Deliver response
                A->>D: Store response/update status
                A->>F: Mark user safe
            else No response
                A->>D: Escalate case as emergency
                A->>F: Show emergency alert
            end
        else Day-time stagnation
            A->>D: Escalate case as high-priority alert
            A->>F: Show high-priority alert
        end
    else Location pattern normal
        A->>A: No escalation
    end

18. Final Implementation Statement
This software design converts FECS from a concept into an implementable application architecture. The application is centered around a FastAPI backend, MQTT-based message ingestion, PostgreSQL alert storage, and a real-time React dashboard with map visualization. The backend stores only valid non-duplicate alerts and triggers ACK only for valid manual alerts through the base station module.
The sequence diagrams define how alerts move through the system and how acknowledgments return to the user module. The architecture diagrams define the software building blocks needed for your team to begin implementation.
