UNCONSCIOUSNESS DETECTION FEATURE - COMPLETE WORKING DOCUMENTATION
==================================================================

1. FEATURE OVERVIEW
-------------------
The Unconsciousness Detection Feature is a proactive safety mechanism designed for the Forest Emergency Communication System (FECS). It mathematically monitors the physical movement of personnel carrying User Modules in the field over a 2-hour window. If a user becomes suspicious stationary (moving less than 10 meters in 2 hours), the system actively challenges them by triggering a physical buzzer on their hardware. If the user fails to acknowledge this challenge within 60 seconds, the software automatically escalates their status to an 'Emergency' on the dashboard.

2. WHY IS THIS FEATURE NECESSARY?
---------------------------------
In remote or hazardous forest environments, traditional SOS systems rely entirely on the victim being conscious and physically capable of pressing an emergency button. 

*   The Blind Spot: If a ranger falls, is attacked by wildlife, or suffers a sudden medical emergency (e.g., snakebite, heart attack) and loses consciousness, they cannot press the SOS button. 
*   The Solution: This feature acts as an automated "Dead Man's Switch." It shifts the system from passive listening to active monitoring. It ensures that an incapacitated ranger is never left behind simply because they couldn't call for help themselves.

3. THE ACK MESSAGE FORMAT (SERVER -> BASE -> USER MODULE)
---------------------------------------------------------
Because the ESP32 parser requires strict adherence to bandwidth and memory constraints, the outbound Acknowledgment (ACK) message from the server strips out heavy JSON and unused timestamp fields, favoring an ultra-lightweight pipe-delimited format.

Final Format Design:
[packet_id] | [device_id] | [ack_type]

Field Breakdown:
1. packet_id: A unique identifier generated by the server (e.g., "ACK_17203..."). Tells the hardware not to process this ping twice if relayed multiple times.
2. device_id: The specific hardware identifier (e.g., "DEV_001"). Ensures that only the targeted ranger's buzzer activates, not the entire forest network.
3. ack_type: A critical single-character flag parsed by the ESP32.
   - 'L' = LED Blink (Quiet confirmation that a normal message was received by the server).
   - 'B' = Buzzer Alarm (Loud challenge fired by the server demanding a manual button press response).

Example Payload (Over MQTT -> LoRa):
ACK_17109283|DEV_001|B

4. CORE FUNCTIONS USED IN IMPLEMENTATION
----------------------------------------

A. Mathematical Calculation: `calculate_distance(lat1, lon1, lat2, lon2)`
   - Location: `backend/app/services/alert_service.py`
   - Purpose: Uses the standard Haversine formula to compute the spherical distance between the latest 2-hour GPS auto-ping and the preceding auto-ping. Accurately determines if the traveler has moved fewer than 10 meters.

B. Data Persistence: `store_alert(...)` & `MonitoringStatus` Model
   - Location: `backend/app/models.py` & `alert_service.py`
   - Purpose: Silently intercepts incoming 'Auto' regular pings and logs the user's coordinates in a dedicated tracking table (`MonitoringStatus`) to provide historical context for the Haversine formula without bloating the primary alert table.

C. Hardware Trigger: `publish_buzzer_ack(device_id, mqtt_client)`
   - Location: `backend/app/services/ack_service.py`
   - Purpose: The exact function that formats the custom 3-field `packet_id|device_id|B` string and pushes it to the `fecs/ack` MQTT topic.

D. The Asynchronous Engine: `monitoring_service.py` / `MonitoringService.run()`
   - Location: `backend/app/services/monitoring_service.py`
   - Purpose: An `asyncio` background loop initialized directly in FastAPI's startup (`main.py`). It wakes up every 60 seconds completely independent of web traffic to execute two core sweeps:
     1. Sweep 1 (The Check): Look for stationary coordinates over 2 hours and trigger `publish_buzzer_ack()`.
     2. Sweep 2 (The Escalation): Look for devices whose buzzers were triggered exactly 60 seconds ago but haven't replied. If found, artificially inject an `Emergency` alert into the database with `source=software` and blast the WebSocket to flash the Operator Dashboard red.
